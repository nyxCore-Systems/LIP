# LIP — Linked Incremental Protocol

**Design Document & Specification v0.2** · Draft · Apache 2.0

---

## Abstract

LIP (Linked Incremental Protocol) is a language-agnostic, open-source protocol for
streaming, incremental, AI-native code intelligence. Version 0.2 significantly expands
the v0.1 foundation based on research into the 2025–2026 ecosystem landscape:

**What changed since v0.1:**

- **MCP integration layer** — LIP is now specified as a first-class MCP Resource
  Provider, reflecting MCP's emergence as the universal agent integration standard
  (donated to Linux Foundation as AAIF in December 2025, >97M monthly SDK downloads).
- **Semantic lifting** — A new Tier 1.5 layer based on RPG-Encoder research
  (arXiv:2602.02084) adds LLM-generated verb-object semantic features to the
  structural symbol graph, enabling natural-language code queries without a separate
  RAG pipeline.
- **SLSA provenance for dependency slices** — Dependency slices now carry SLSA v1.1
  Build Level 2 provenance attestations, addressing supply-chain security at the
  protocol level.
- **Agent2Agent (A2A) gateway** — LIP daemons can coordinate through Google's A2A
  protocol for multi-repo and multi-service intelligence across agent swarms.
- **Kythe compatibility layer** — With Google Kythe's US team dissolved in April 2024,
  a significant open-source vacuum exists. LIP provides a Kythe graph importer to
  ease migration for existing Kythe consumers.
- **GraphRAG query surface** — The query API is extended with a graph-native subgraph
  retrieval interface aligned with recent GraphRAG research (CodeRAG, RepoGraph).

## 1. Motivation & Prior Art Analysis

### 1.1 The LSP Gap

The Language Server Protocol (v3.17, Microsoft) solved the M×N IDE-language matrix
problem but was designed exclusively for interactive, file-scoped intelligence:

- State is volatile — lost on server restart.
- Cross-file queries are expensive or absent.
- Incremental text sync has known drift bugs: each LSP client (nvim-lsp, eglot,
  helix, sublime) handles document versions differently, causing state mismatches that
  are extremely hard to debug.
- No concept of persistent, shareable artifacts.
- LSP operations are largely scoped to individual files — AI agents require
  additional code graph indexing layers that are not standardized.

### 1.2 The SCIP Bottleneck

SCIP improved on LSIF by replacing opaque global IDs with human-readable symbol URIs
and using Protobuf for efficient encoding. However:

- Globally incrementing IDs in LSIF — and the batch model inherited by SCIP —
  make it difficult to update an existing index with new information for only a subset
  of documents.
- Full re-indexing is required on every meaningful change.
- No team cache sharing — every machine re-indexes the same dependencies.
- Not designed for streaming or agent consumption.

### 1.3 The Kythe Vacuum

Google Kythe — the most ambitious cross-language code indexing system, grown from
Google's internal Grok tool — lost its entire US-based development team in April 2024
as Google moved the project to a maintenance-only India team. This leaves a major
open-source vacuum in the cross-language, cross-repo code intelligence space.
Kythe's hub-and-spoke model (O(L+C+B) instead of O(L×C×B)) was architecturally
elegant, but its steep learning curve, bespoke schema, and requirement to instrument
compilers limited adoption outside Google. LIP inherits Kythe's cross-language
graph philosophy while eliminating the instrumented-build requirement.

### 1.4 The MCP Layer Gap

The code intelligence landscape in 2025–2026 is fragmented: tools like GitNexus,
CodeGraphContext, Sourcegraph Cody, Greptile, and Repomix solve overlapping problems
from different angles. MCP provides the transport layer but not the semantic layer.
The winners will be whoever defines the standard semantic layer for code intelligence.

LIP aims to be that semantic layer — a protocol that sits above MCP the same way LSP
sits above JSON-RPC.

## 2. Ecosystem Context (2025–2026)

Understanding the current landscape is essential to positioning LIP correctly.

### 2.1 MCP as universal transport

MCP was donated to the Agentic AI Foundation (AAIF) under the Linux Foundation in
December 2025, co-founded by Anthropic, Block, and OpenAI. It is now the de facto
protocol for connecting AI systems to external tools and data. By early 2026,
MCP is natively supported by virtually every major AI development tool: Claude Code,
Cursor, VS Code (via Copilot), JetBrains, Zed, and Sourcegraph.

LIP's design decision: expose a native MCP server surface so any MCP-compatible agent
can query the LIP intelligence graph without a custom integration.

### 2.2 Agentic coding as the primary consumer

Advances in structured code representations — ASTs, graph-based IRs, semantic
embeddings — offer a foundation for more powerful agent reasoning. Combining LLMs with
graph neural networks or neuro-symbolic systems could improve generalization and support
cross-language, cross-target understanding.

AI coding agents routinely break code they do not understand structurally — editing a
function without knowing 47 others depend on it. Code intelligence tools solve this by
giving agents codebase awareness before they edit. LIP's blast radius and impact
preview APIs are designed precisely for this use case.

### 2.3 Semantic lifting as a new tier

Recent research (RPG-Encoder, arXiv:2602.02084) demonstrates that enriching a
structural code graph with LLM-generated semantic annotations — verb-object features
like "validate user credentials" or "serialize config to disk" — dramatically improves
natural-language code retrieval. The graph is saved and committed to the repo,
giving all collaborators and AI tools instant semantic search without rebuilding.

LIP v0.2 formalizes this as Tier 1.5 in the confidence model.

### 2.4 Supply chain security as table stakes

SLSA (Supply-chain Levels for Software Artifacts) is the industry standard for
securing software build processes. Levels 1–4 apply to source code, build systems,
packaging, and dependencies. SLSA Level 2 adds cryptographic verification of build
sources and authenticated build environments.

LIP's dependency slices, which are pre-built index artifacts for external packages,
are now first-class supply-chain artifacts and must carry SLSA provenance attestations.

### 2.5 The GraphRAG convergence

Structure-based RAG models a repository as a graph where code nodes represent
snippets (modules, classes, functions) and edges represent relationships (imports,
invocations, inheritance). This approach improves repository-level code generation
significantly over naive RAG. LIP's query graph is directly compatible with this
pattern — the `lip.query.subgraph` API is the GraphRAG retrieval interface.

## 4. Core Design Principles (updated)

### 4.1 Lazy Query Graph (Salsa-inspired)

*(unchanged from v0.1 — see original spec)*

Every derived knowledge artifact is a pure function in a directed acyclic query graph
with a global revision counter. Changes are lazy: the counter increments O(1),
re-computation happens only when queried, with early cutoff when outputs are stable
despite changed inputs.

When computing a top-level query after a change, Salsa does two graph traversals:
first forward, flooding down to the inputs; if a changed input is hit, it floods
backward propagating the change but stopping when a query's output is unchanged despite
the change — the "early cutoff" that makes incremental computation efficient.

**Core invariant**: Changing a private function body never invalidates the exported API
surface queries of its containing module.

### 4.2 Blast Radius Indexing

LIP maintains a reverse-dependency graph. When a file is saved:

1. Compute API surface diff (exported symbols only).
2. If surface unchanged: no graph invalidation.
3. If surface changed: walk reverse dep graph, re-verify only affected files.

Complexity: `O(Δ + depth(reverse_deps))` vs. SCIP's `O(N_total)`.

### 4.3 Four-Tier Confidence Model

v0.2 introduces **Tier 1.5 — Semantic Lift**, between Tree-sitter and compiler
verification. This tier applies LLM-generated semantic annotations to the structural
symbol graph.

#### Tier 1 — Syntactic <Badge variant="gray">score 1–30</Badge>

| Property | Value |
|---|---|
| Engine | Tree-sitter CST parser |
| Latency | < 10 ms |
| Scope | Current file + dirty buffers |
| Output | Symbol positions, kinds, local refs |
| UI signal | Dotted underline (verifying) |

Tree-sitter preserves every token and boundary, making it perfect for retrieval
and for mapping results back to the exact line and column in source code — essential for
editor integrations and targeted context for LLMs.

#### Tier 1.5 — Semantic Lift <Badge variant="teal">score 31–55</Badge>

*New in v0.2.*

| Property | Value |
|---|---|
| Engine | Background LLM call (local or remote, configurable) |
| Latency | 1–10 s per function (batched, async) |
| Scope | Function/class level, repo-wide |
| Output | Verb-object feature tags, intent annotations, functional domain labels |
| UI signal | Hover shows semantic summary |

Based on RPG-Encoder (arXiv:2602.02084): each function or class in the symbol graph
is annotated with a short, LLM-generated verb-object description (e.g.
"validate user credentials", "serialize config to disk"). These annotations:

- Enable **natural-language workspace search** without a separate embedding pipeline.
- Provide **functional domain clustering** (grouping symbols by what they do, not
  where they live).
- Persist to the query graph and are shared through the team cache — a team member
  who lifted `auth.dart` means no one else needs to.

The lifting operation is incremental: only symbols changed since the last lift are
re-annotated.

```flatbuffers
// New field in SymbolInfo (v0.2)
semantic_features: [string];  // e.g. ["validate credentials", "check expiry"]
functional_domain: string;    // e.g. "authentication", "data serialization"
lift_confidence:   uint8;     // 1–100, from the LLM response
lift_model:        string;    // e.g. "claude-sonnet-4-20250514" or "local/qwen2.5"
lift_timestamp_ms: int64;
```

#### Tier 2 — Local Verified <Badge variant="purple">score 56–85</Badge>

| Property | Value |
|---|---|
| Engine | Incremental compiler / language-specific analyzer |
| Latency | 200–500 ms after save |
| Scope | Full local repo, incremental blast radius |
| Output | Compiler-accurate types, cross-file refs |
| UI signal | Dotted underline removed |

#### Tier 3 — Global Anchor <Badge variant="amber">score 86–100</Badge>

| Property | Value |
|---|---|
| Engine | Federated CAS registry pull |
| Latency | Instant (downloaded once) |
| Scope | External deps (npm, cargo, pub, pip, go) |
| Output | Pre-built, SLSA-attested symbol graph |
| UI signal | Lock icon on hover |

### 4.4 Federated Dependency Slicing with SLSA Provenance

v0.2 adds supply-chain security to dependency slices. Every slice published to the
global registry must carry:

1. **Content hash** — SHA-256 of the slice blob (integrity).
2. **SLSA v1.1 Build Level 2 provenance attestation** — signed by the registry's
   Ed25519 key, recording: builder ID, build inputs (package source + version),
   build timestamp, output digest.
3. **SPDX license identifier** — included in the slice metadata.

```flatbuffers
// Extended DependencySlice (v0.2)
table DependencySlice {
  manager:            string;
  package_name:       string;
  version:            string;
  package_hash:       string;
  content_hash:       string;
  symbols:            [SymbolInfo];
  slice_url:          string;
  built_at_ms:        int64;
  // ── New in v0.2 ────────────────────────────────────────────
  spdx_license:       string;      // "MIT", "Apache-2.0", etc.
  slsa_provenance:    SLSAProvenance;
  semantic_features_included: bool; // whether Tier 1.5 lift is included
}

table SLSAProvenance {
  builder_id:         string;      // registry builder identity
  build_type:         string;      // "https://lip-protocol.dev/build/v1"
  source_uri:         string;      // canonical package source URL
  source_digest:      string;      // SHA-256 of source tarball
  invocation_id:      string;      // build job ID
  built_at_ms:        int64;
  signature:          bytes;       // Ed25519 signature over above fields
  signing_key_id:     string;      // key fingerprint for verification
}
```

Clients MUST verify the SLSA signature before mounting a slice. A slice with an invalid
or missing signature is rejected and re-fetched or flagged for manual review.

### 4.5 Merkle Sync & State Integrity

*(unchanged from v0.1)*

The full repository state is a Merkle tree. Client and daemon reconcile via
binary-tree search over mismatched hashes. No full re-index on divergence.

### 4.6 Semantic Lifting

The lifting process is asynchronous and non-blocking:

1. After Tier 1 indexing, the daemon queues all new/changed symbols for lifting.
2. A configurable LLM backend processes the queue in batches (default: 50 symbols
   per call, 8000 token budget per batch).
3. Results are written into the query graph as Tier 1.5 annotations.
4. Delta events are streamed to IDE clients with the new `semantic_features` populated.
5. Team cache: lifted annotations are included in dependency slices. Local symbols'
   lift results are synchronized through the team registry using the same CAS
   mechanism as dependency slices.

**LLM backend options** (configurable in `lip.toml`):

```toml
[lifting]
backend = "anthropic"           # "anthropic" | "openai" | "local" | "disabled"
model = "claude-haiku-4-5-20251001"  # fast, cheap for bulk annotation
batch_size = 50
max_tokens_per_batch = 8000
local_endpoint = ""             # for "local" backend: ollama/LM Studio URL
```

Lifting is entirely optional and can be disabled. The protocol functions without it —
Tier 1.5 symbols simply have empty `semantic_features`.

## 5. Wire Format — FlatBuffers Schema

### 5.1 Full schema (`lip.fbs` v0.2)

```flatbuffers
// lip.fbs — LIP FlatBuffers Schema
// Version: 0.2.0
// License: Apache 2.0
// Changes from v0.1:
//   - Added SemanticFeatures to SymbolInfo
//   - Added SLSAProvenance to DependencySlice
//   - Added McpResourceDescriptor
//   - Added A2AEndpoint to EventStream
//   - confidence_score tiers adjusted (4 tiers, max 100)

namespace lip;

// ─── Envelope ────────────────────────────────────────────────────────────────

table EventStream {
  deltas:          [Delta];
  schema_version:  uint16 = 2;        // bumped from 1
  emitter_id:      string;
  timestamp_ms:    int64;
  a2a_source:      A2AEndpoint;       // new: identifies the originating agent
}

table Delta {
  action:       Action;
  commit_hash:  string;
  document:     Document;
  symbol:       SymbolInfo;
  slice:        DependencySlice;
  mcp_resource: McpResourceDescriptor; // new: MCP resource descriptor
}

enum Action : byte { Upsert = 0, Delete = 1 }

// ─── Document ────────────────────────────────────────────────────────────────

table Document {
  uri:            string;
  content_hash:   string;
  language:       string;
  occurrences:    [Occurrence];
  symbols:        [SymbolInfo];
  merkle_path:    string;
  // New in v0.2: functional domain inferred from lifting
  functional_domain: string;
}

// ─── Occurrence ──────────────────────────────────────────────────────────────

table Occurrence {
  symbol_uri:       string;
  range:            Range;
  confidence_score: uint8;      // 1=Tier1 syntactic … 100=Tier3 global anchor
  role:             Role;
  override_doc:     string;
}

table Range {
  start_line: int32;
  start_char: int32;
  end_line:   int32;
  end_char:   int32;
}

enum Role : byte {
  Definition      = 0,
  Reference       = 1,
  Implementation  = 2,
  TypeBinding     = 3,
  ReadAccess      = 4,
  WriteAccess     = 5,
}

// ─── Symbol ──────────────────────────────────────────────────────────────────

table SymbolInfo {
  uri:               string;
  display_name:      string;
  kind:              SymbolKind;
  documentation:     string;
  signature:         string;
  confidence_score:  uint8;
  relationships:     [Relationship];
  // ── AI extension slots ───────────────────────────────────────────────────
  runtime_p99_ms:    float32 = -1;
  call_rate_per_s:   float32 = -1;
  taint_labels:      [string];
  blast_radius:      uint32 = 0;
  // ── New in v0.2: Semantic Lift (Tier 1.5) ───────────────────────────────
  semantic_features: [string];      // verb-object annotations
  functional_domain: string;        // high-level domain label
  lift_confidence:   uint8 = 0;     // 0 = not lifted
  lift_model:        string;        // model used for lifting
  lift_timestamp_ms: int64 = 0;
  // ── Token budget hint for agents ────────────────────────────────────────
  token_estimate:    uint32 = 0;    // estimated LLM tokens for this symbol's context
}

enum SymbolKind : byte {
  Unknown=0, Namespace=1, Class=2, Interface=3, Method=4, Field=5,
  Variable=6, Function=7, TypeParameter=8, Parameter=9, Macro=10,
  Enum=11, EnumMember=12, Constructor=13, TypeAlias=14,
  // New in v0.2
  Extension=15,    // Dart extensions, Swift extensions, etc.
  Mixin=16,        // Dart mixins
  Widget=17,       // Flutter widget (Dart-specific, first-class)
}

table Relationship {
  target_uri:          string;
  is_implementation:   bool;
  is_reference:        bool;
  is_type_definition:  bool;
  is_override:         bool;
  // New in v0.2
  is_widget_child:     bool;   // Flutter widget tree relationship
  edge_label:          string; // Kythe-compatible edge label for importers
}

// ─── Dependency Slice ────────────────────────────────────────────────────────

table DependencySlice {
  manager:                    string;
  package_name:               string;
  version:                    string;
  package_hash:               string;
  content_hash:               string;
  symbols:                    [SymbolInfo];
  slice_url:                  string;
  built_at_ms:                int64;
  // New in v0.2
  spdx_license:               string;
  slsa_provenance:            SLSAProvenance;
  semantic_features_included: bool;
}

table SLSAProvenance {
  builder_id:    string;
  build_type:    string;       // "https://lip-protocol.dev/build/v1"
  source_uri:    string;
  source_digest: string;
  invocation_id: string;
  built_at_ms:   int64;
  signature:     [byte];       // Ed25519 signature
  signing_key_id:string;
}

// ─── MCP Resource Descriptor (new in v0.2) ───────────────────────────────────

/// Describes a LIP symbol or subgraph as an MCP Resource.
/// Allows MCP clients to discover and retrieve LIP intelligence natively.
table McpResourceDescriptor {
  uri:           string;    // mcp://lip/{symbol-uri-encoded}
  name:          string;    // human-readable resource name
  description:   string;    // includes semantic_features if available
  mime_type:     string;    // "application/lip+flatbuffers" or "application/json"
  size_bytes:    uint32;
}

// ─── A2A Gateway (new in v0.2) ───────────────────────────────────────────────

/// Identifies an agent endpoint in an A2A-coordinated multi-agent system.
table A2AEndpoint {
  agent_id:      string;    // unique agent identifier
  service_url:   string;    // A2A service URL
  capabilities:  [string];  // e.g. ["lip/query", "lip/index", "lip/taint"]
}

root_type EventStream;
```

### 5.2 Schema evolution

- Fields appended with defaults are backward-compatible.
- `schema_version: 2` — v0.1 clients may ignore unknown fields.
- `SymbolKind.Extension`, `.Mixin`, `.Widget` are Dart/Flutter-specific first-class
  kinds. Other languages that have no widget/mixin concept will never emit them.
- The `edge_label` field on `Relationship` is populated only by the Kythe importer.

## 7. Protocol Lifecycle

*(updated to include MCP handshake and lifting phases)*

### 7.1 Phase 0 — Daemon startup

Daemon loads persisted query graph, connects to configured team registry.

### 7.2 Phase 1 — Handshake & manifest

```
Client → Daemon:  ManifestRequest {
  repo_root:       string,
  merkle_root:     string,
  dep_tree_hash:   string,
  lip_version:     string,
  mcp_enabled:     bool,       // new
  lifting_config:  LiftConfig, // new
}

Daemon → Client:  ManifestResponse {
  cached_merkle_root:  string,
  missing_slices:      [string],
  indexing_state:      IndexingState,
  mcp_server_url:      string,  // new: local MCP endpoint
  lifting_queue_size:  uint32,  // new: pending symbols to lift
}

enum IndexingState { Cold, WarmPartial, WarmFull }
```

### 7.3 Phase 2 — Shallow parse (< 30 s)

Tree-sitter over all tracked files → Tier 1 symbols. Dependency slices fetched
in parallel, SLSA attestations verified before mounting.

### 7.4 Phase 2.5 — Semantic lift (new, async)

Background LLM calls annotate symbols with Tier 1.5 features. Lifting runs
concurrently with Phase 3 and does not block compiler verification. Lifted symbols
emit `Delta.Upsert` events with `semantic_features` populated.

### 7.5 Phase 3 — Background verification (30 s – 5 min)

Incremental compiler processes blast radius in priority order (most-imported modules
first). Tier 1 → Tier 2 upgrades streamed to client.

### 7.6 Phase 4 — Steady state

On file save: API surface diff → blast radius walk → targeted re-verification.
Lifting re-queues only changed symbols. Typical latency: 200–800 ms.

### 7.7 Query API (updated)

```
# v0.1 queries (unchanged)
lip.query.definition(uri, position) → SymbolInfo
lip.query.references(symbol_uri, limit?) → [Occurrence]
lip.query.hover(uri, position) → HoverResult
lip.query.blast_radius(symbol_uri) → BlastRadiusResult
lip.query.subgraph(symbol_uri, depth) → SymbolGraph
lip.query.taint(symbol_uri) → [TaintPath]
```

## 8. MCP Integration

*New in v0.2.*

LIP exposes a native MCP server surface, making it a first-class citizen in the MCP
ecosystem. MCP takes inspiration from the Language Server Protocol for standardizing
how to add support for programming languages across a whole ecosystem of development
tools — in a similar way, LIP-over-MCP standardizes how code intelligence integrates
into the ecosystem of AI agent applications.

### 8.1 LIP as MCP Resource Provider

The LIP daemon runs an embedded MCP server on a local port (default: `3714`).

**Resources** (read-only, streamed via MCP `resources/list` and `resources/read`):

```
mcp://lip/symbol/{uri-encoded}          → SymbolInfo (JSON or FlatBuffers)
mcp://lip/subgraph/{uri-encoded}/{depth} → SymbolGraph
mcp://lip/document/{file-uri-encoded}   → Document with all occurrences
mcp://lip/domain/{domain-name}          → [SymbolInfo] for a functional domain
mcp://lip/blast-radius/{uri-encoded}    → BlastRadiusResult
```

**Tools** (callable via MCP `tools/call`):

```json
{
  "tools": [
    {
      "name": "lip_semantic_search",
```

## 9. Transport & IPC

*(largely unchanged from v0.1, with MCP addition)*

### 9.1 IDE ↔ Daemon (local)

Unix domain socket (Linux/macOS) or named pipe (Windows) + shared mmap for zero-copy
FlatBuffers reads.

### 9.2 Daemon ↔ Registry (remote)

gRPC streaming over TLS for slice downloads. HTTP/HTTPS blob endpoints for CDN

### 9.3 Daemon ↔ CI (incremental push)

CI runners emit `EventStream` deltas per commit. No full `.scip` files.

### 9.4 MCP server (new)

Embedded HTTP/SSE server for MCP protocol. Uses JSON-RPC 2.0 over HTTP with
Server-Sent Events for streaming responses, matching MCP spec November 2025.

### 9.5 A2A gateway (new)

Optional gRPC endpoint for inter-daemon coordination via Google's Agent2Agent protocol.
See §12.

## 10. Intelligence Extensions (updated)

### 10.1 Blast radius analysis (unchanged)

Returns direct/transitive dependent counts and affected files/services.

### 10.2 Taint tracking (unchanged)

Propagates `taint_labels` through data flow graph. Returns paths to unsafe sinks.

### 10.3 Runtime telemetry overlay (unchanged, now available via MCP)

`runtime_p99_ms` and `call_rate_per_s` from OpenTelemetry/Datadog integration.

### 10.4 Dead code detection (unchanged)

Zero-reference + zero-runtime-call symbols.

### 10.5 Natural-language semantic search (new in v0.2)

```
lip.query.semantic_search("validate payment before processing") → [
  {uri: "lip://local/myapp/lib/payments.dart#PaymentService.validate", ...},
  {uri: "lip://local/myapp/lib/orders.dart#OrderService.checkPaymentStatus", ...},
]
```

Powered by Tier 1.5 semantic features. Does not require a vector database or embedding
pipeline — the LLM-generated feature strings are indexed with a simple inverted index
on the daemon side. For higher precision, an embedding backend can be configured:

```toml
[semantic_search]
backend   = "inverted_index"   # "inverted_index" | "embedding"
embed_url = ""                 # if backend = "embedding"
embed_dim = 768
```

### 10.6 Functional domain map (new in v0.2)

```
lip.query.functional_domain("authentication") → {
  symbols: [AuthService, UserTokenStore, OAuthProvider, ...],
  files:   ["lib/auth.dart", "lib/oauth.dart", ...],
  domain_summary: "Handles user authentication including OAuth2 flows,
                   token validation, and session management."
}
```

The domain map is generated from the lifting process and provides a high-level
architectural overview of the codebase — useful for onboarding new agents and developers.

### 10.7 Widget tree intelligence (new, Dart/Flutter-specific)

For Flutter projects, LIP indexes the widget tree as a first-class graph structure.
`is_widget_child` edges on `Relationship` encode parent-child widget relationships.
This enables:

- Widget tree queries: "find all Scaffold descendants of HomeScreen"
- Impact analysis on widget rebuilds
- Hover showing the widget tree path from root

## 11. AI & Agent Integration (updated)

### 11.1 Semantic subgraph queries (updated)

The `lip.query.subgraph` response now includes `semantic_features` and `token_estimate`
on all returned symbols, enabling agents to make informed context-window decisions.

```
lip.query.subgraph(
  symbol_uri:      "lip://local/myapp/lib/checkout.dart#CheckoutService",
  depth:           2,
  include_types:   true,
  include_callers: true,
  include_callees: true,
  max_tokens:      4000,   // new: stops expanding graph when budget is reached

  edges:           [Edge],

}
```

### 11.2 Streaming context for RAG (updated)

The streaming context endpoint now sorts by confidence tier first, then by semantic
relevance. Tier 3 (global anchor, pre-verified) symbols are always returned before
lower-confidence local symbols.

### 11.3 Change impact preview (updated)

Now includes semantic impact: if a changed function has a `functional_domain` label,
the response includes a natural-language summary of what parts of the system will be
affected.

## 12. A2A Gateway

*New in v0.2.*

For teams running multiple LIP daemons across services or microservices in a monorepo,
the A2A gateway enables cross-daemon intelligence coordination.

Two open protocols have emerged as the backbone of the agentic ecosystem: MCP
standardizes how AI models interact with external tools and data, while A2A (launched
by Google with 50+ enterprise partners) handles asynchronous agent-to-agent
communication, letting agents discover, negotiate, and collaborate with each other.

### 12.1 Use case

In a monorepo with separate LIP daemons for `service-auth`, `service-payments`, and
`service-orders`, a blast radius query on `AuthService.verifyToken` must cross service
boundaries. The A2A gateway federates these queries.

### 12.2 Gateway protocol

Each LIP daemon with A2A enabled publishes an **Agent Card** at:
`GET /.well-known/lip-agent.json`

```json
{
  "id": "lip-daemon-auth-service",
  "name": "LIP Daemon — auth service",
  "url": "https://auth.internal:3715",
  "capabilities": ["lip/query", "lip/index", "lip/blast-radius"],
  "lip_version": "0.2.0",
  "repo_root": "/workspace/service-auth",
  "languages": ["dart"]
}
```

Cross-daemon queries use the A2A task model: the querying daemon creates a task on the
target daemon, waits for progress events, and assembles the merged result.

### 12.3 Configuration

```toml
[a2a]
enabled = false
port    = 3715
peers   = [
  "https://payments.internal:3715",
  "https://orders.internal:3715",
]
```

### 13.1 LIP-to-LSP bridge

Unchanged from v0.1. LIP responses are translated to LSP JSON-RPC on demand.

### 13.2 SCIP importer

Unchanged from v0.1. `lip import --from-scip ./index.scip` bootstraps the LIP cache

### 13.3 SCIP exporter

`lip export --to-scip ./index.scip` for downstream SCIP consumers.

### 13.4 Kythe importer (new in v0.2)

```bash

```

Reads Kythe's flat entry format and converts:
- Kythe VNames → LIP URIs (`lip://kythe/{corpus}/{path}#{signature}`)
- Kythe edges → LIP `Relationship` with `edge_label` populated
- Kythe anchor facts → LIP `Occurrence` with `confidence_score: 75`

Kythe's semantic richness (call graph, override chains, cross-language links) is fully
preserved. This provides a migration path for teams previously using Kythe with Google
services or open-source tooling.

### 13.5 MCP bridge (new in v0.2)

Any MCP-compatible client can discover LIP resources at `mcp://lip/` without requiring
a LIP-native client. See §8.

## 14. Security & Supply Chain (updated)

### 14.1 Dependency slice integrity (updated)

Slices are verified via:
1. `content_hash` (SHA-256) — integrity.
2. `slsa_provenance.signature` (Ed25519) — authenticity and provenance.
3. Optional: Sigstore transparency log inclusion check for public registry slices.

The SLSA Level 2 requirement means: build inputs are cryptographically verified, the
build environment is authenticated, and provenance is digitally signed. This prevents
a malicious registry from substituting a tampered slice.

### 14.2 Symbol URI validation (unchanged)

URIs validated against grammar. Path traversal sequences, null bytes, non-UTF-8
content rejected.

### 14.3 MCP security

The embedded MCP server binds to `127.0.0.1` by default. Token auth is available
for team scenarios where the MCP server is exposed over a private network. The server
never executes arbitrary code — it only exposes read queries on the LIP graph.

MCP's security model requires that hosts obtain explicit user consent before
invoking any tool and that tool descriptions be treated as untrusted unless obtained
from a trusted server. LIP's MCP server is always a trusted local server — it
runs on the developer's own machine and never makes outbound tool calls.

### 14.4 Lifting confidentiality

Semantic lifting requires sending code to an LLM backend. For teams with confidentiality
requirements:

- Use `backend = "local"` with a local model (Ollama, LM Studio).
- Use `backend = "disabled"` to opt out entirely.
- The `lifting_scope` config allows limiting lifting to non-sensitive files:

```toml
[lifting]
scope_include = ["lib/**", "src/**"]
scope_exclude = ["lib/secrets/**", "**/*.key"]
```

### 14.5 Taint label trust (unchanged)

Taint labels are advisory. They are propagated by the daemon but never enforced at
the protocol level.

### 14.6 Shared memory safety (unchanged)

mmap region is `MAP_PRIVATE` on reader side, bounds-checked before access.

## Appendix A — Rationale for FlatBuffers over Protobuf

*(unchanged from v0.1)*

FlatBuffers allows zero-copy reads directly from mmap'd buffers — the IDE can query
specific fields by offset without deserializing the full message. SCIP's Protobuf
requires full allocation on every parse. For high-frequency symbol queries (e.g., per
cursor-move hover), this makes a measurable difference.

FlatBuffers serialization (write path) is slower, but LIP's write path is the
background daemon — the read path (IDE querying symbols) is the latency-critical path.

## Appendix B — Prior Art Deep Dive

### LSP (Language Server Protocol)

Created by Microsoft in 2016. JSON-RPC 2.0 transport. v3.17 is current (2022).
Solved the M×N language-IDE integration problem. Foundation of modern IDE intelligence.
Limitation: file-scoped, volatile state, no persistent cross-repo index.

### LSIF (Language Server Index Format)

Microsoft 2019. Graph encoding with opaque global IDs. Superseded by SCIP.
Limitation: large file sizes, ID ordering constraints, difficult incremental updates.

### SCIP (Symbolic Code Intelligence Protocol)

Sourcegraph 2022. Protobuf-based, human-readable symbol URIs. Significant improvement
over LSIF. Still batch-oriented, no team cache sharing, no incremental model.

### Kythe

Google 2015 (open-sourced). Hub-and-spoke model reducing L×C×B language-client-
build integration to O(L+C+B). Elegant architecture, powerful cross-language graph
schema. Limitations: bespoke VName scheme, instrumented build requirement, steep
learning curve. Maintenance collapsed after April 2024 US team layoffs.
The dedicated Google team was replaced with an India-based maintenance team.

### Glean (Meta)

Internal code indexing system with a Datalog-like query layer. Open-sourced 2021.
Strong for cross-language queries, used at Meta scale. Not widely adopted externally
due to Datalog learning curve and infrastructure requirements.

### SemanticDB (Scala ecosystem)

Heavily influenced SCIP's symbol design. Human-readable symbol syntax, clean schema.
Scala-specific but the URI design generalized well into SCIP and LIP.

### Salsa (Rust incremental computation)

Used by rust-analyzer and the Rust compiler. The definitive reference for
lazy, incremental query computation. LIP's query graph design is directly inspired
by Salsa's durability/early-cutoff model.

### RPG-Encoder (arXiv:2602.02084)

2026 research from Luo et al. Combines Tree-sitter structural parsing with LLM-based
semantic lifting. Key insight: verb-object semantic features ("validate user credentials")
dramatically improve natural-language code search without requiring vector embeddings.
Introduced the idea of committing the semantic graph to the repo for team sharing.
LIP's Tier 1.5 is a protocol-level formalization of this approach.
